<objective>
Convert the async engine from NullPool to a real connection pool (PERF-02), add DB indexes on hot query columns via an Alembic migration (PERF-01), and replace the timestamp-based order code generator with a collision-safe UUID-based version (PERF-05).

Purpose: Without pooling, every request opens a new asyncpg connection — kills throughput. Without indexes, slug lookup, category filter, and order list queries do sequential scans. The current `generate_order_code()` returns the same string for two simultaneous checkouts in the same second.
Output: Pooled engine, indexed schema, unique order codes.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Concurrent client → orders router | Two simultaneous checkouts can race on order_code uniqueness |
| App → DB | Connection pool exhaustion under load can deny service |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-08 | Tampering | Two orders with the same order_code allow guest-lookup confusion (one user finds another's order) | mitigate | UUID4 hex prefix gives 48 bits of entropy; DB-level UNIQUE constraint via Index(unique=True) on orders.order_code is the final guard |
| T-01-09 | Denial of Service | NullPool causes per-request connection storms; DB rejects connections at concurrency >100 | mitigate | Switch to default AsyncAdaptedQueuePool with pool_size=10, max_overflow=20 — bounded but reusable |
| T-01-10 | Denial of Service | Sequential scans on Product / Order tables degrade as data grows | mitigate | Indexes on hot columns ensure logarithmic lookup |
| T-01-11 | Information Disclosure | Order code timestamp leaks order timing/sequence to attackers | mitigate | Random UUID hex reveals nothing about order timing or volume |
</threat_model>
