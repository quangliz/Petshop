<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → FastAPI | Untrusted user input crosses CORS + rate-limit boundary |
| Reverse proxy → FastAPI | X-Forwarded-For header is attacker-controllable; only trustable if proxy strips client-supplied versions |
| Env file → process | SECRET_KEY value comes from .env or environment; if unset, JWT signing is broken/insecure |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | Cross-origin request from attacker domain with credentials | mitigate | CORS allow_origins set to settings.ALLOWED_ORIGINS (explicit list); allow_credentials only effective for listed origins per CORS spec |
| T-01-02 | Tampering | JWT signing with empty/default SECRET_KEY allows forging tokens | mitigate | Lifespan raises RuntimeError if SECRET_KEY is empty or equals _SECRET_KEY_SENTINEL — app cannot start in that state |
| T-01-03 | Denial of Service | Brute-force credential stuffing on /auth/login | mitigate | @limiter.limit("5/minute") per IP via SlowAPI + Redis storage; returns 429 |
| T-01-04 | Denial of Service | Email-bomb via /auth/forgot-password | mitigate | @limiter.limit("3/minute") per IP — tighter than login |
| T-01-05 | Information Disclosure | request.client.host AttributeError leaks 500 stacktrace in payments | mitigate | Guarded access with x-forwarded-for fallback then "127.0.0.1" sentinel |
| T-01-06 | Spoofing | Attacker spoofs X-Forwarded-For to bypass rate limits or audit logs | accept | Rate-limit key uses SlowAPI's get_remote_address (which respects ASGI scope, not headers); X-Forwarded-For is only used in payments IP audit field where this risk is acceptable for a thesis demo. Documented in Phase 3 deferred for production. |
| T-01-07 | Repudiation | Login attempts not logged | accept | Out of Phase 1 scope; FEAT-style audit logging deferred |
</threat_model>
